# 通用非 Android Feature 技术方案生成器

## 目标

根据用户提供的需求资料、代码上下文、接口说明、架构约束或文字描述，生成一份用于非 Android 软件工程单个新特性/版本需求/功能改造评审的技术方案文档。文档应能直接复制到项目空间继续完善，结构严格遵循 `references/template.md`。

## 触发场景

仅在任务同时满足以下条件时使用本 skill：

1. 需求对象属于非 Android 软件工程：后端服务、Web 前端、iOS、桌面端、云平台、数据平台、AI/ML 服务、基础设施、中台系统、SDK/库、CLI 工具、企业内部系统等。
2. 用户目标是编写单个 feature 的技术方案、技术设计、方案评审材料，或按通用技术方案模板补全文档。

典型触发场景：

- 用户要求为后端/Web/iOS/桌面端/云平台/数据平台新 feature 写技术方案。
- 用户提供 PRD、Jira、Confluence、需求截图、接口文档、系统架构或代码路径，希望生成方案评审文档。
- 用户要求按“通用技术方案模版/模板”填写方案。
- 用户让你先读代码、理解调用链、接口契约或部署拓扑，再输出 feature 技术设计。
- 用户要求补充业务流程图、数据流程图、技术架构拓扑图、异常边界场景、风险和解决措施。

不要在以下情况使用本 skill：

- 需求对象属于 Android 开发，应使用 Android 技术方案类 skill。
- 用户只要求概要设计/HLD，应使用通用 HLD 类 skill。
- 用户要求拆任务、估算工作量、多人排期、需求矩阵或甘特图，应使用通用需求矩阵类 skill。
- 上下文无法判断是否非 Android 工程时，先向用户确认，不要仅凭“技术方案/模板”等关键词触发。

## 工作流程

### 1. 收集和阅读输入

先认真阅读用户提供的所有资料。资料可能包括：

- 需求文本、PRD、Jira、Confluence、验收标准、会议纪要。
- 代码仓库路径、模块名、服务名、接口定义、调用链、部署配置。
- UI 原型、流程截图、日志、接口协议、数据结构、系统限制。
- 已有概要设计、旧技术方案、架构图、接口文档或相关历史文档。

如果用户给的是本地工程或文件路径，先用工具读取文件、搜索关键调用链，再写方案。不要只根据文件名或猜测生成内容。

### 2. 提取关键信息

从资料中整理以下信息：

- 功能目标：这次 feature 要解决什么问题、用户侧或系统侧变化是什么。
- 项目背景：项目优先级、上线范围、目标版本、业务价值，无法确定则标注“待确认”。
- 系统背景：基于哪个产品、服务、端、平台、分支、模块或运行环境开发。
- 需求来源：PRD、Jira、Confluence、接口文档链接；缺失时保留“待补充”。
- 涉及模块：前端页面、后端服务、API 网关、数据库、消息队列、缓存、任务调度、第三方服务、SDK/库、基础设施等。
- 核心流程：用户操作路径、服务处理链路、跨系统/跨服务调用。
- 数据流：输入、状态、缓存、持久化、网络/API/消息调用、返回结果。
- 接口依赖：新增接口、复用接口、回调、鉴权、配置开关、灰度、埋点等。
- 异常和边界：网络超时、并发冲突、重复提交、幂等、限流、数据不一致、服务降级、权限不足、版本兼容等。
- 风险和措施：方案劣势、不可控依赖、兼容性、性能、稳定性、安全、回滚策略和验证方法。

### 3. 加载模板并保持章节完整

读取 `references/template.md` 作为章节模板。生成文档时保持模板中的必填章节完整，不要删除已列出的一、二、三级标题。可以根据项目需要新增章节，但新增内容应放在模板既有结构之后或相关小节内，不要破坏主结构。

必填章节包括：

- 1. 背景
- 1.1 项目背景
- 1.2 系统背景
- 2. 需求描述
- 3. 技术方案
- 3.1 方案描述
- 3.2 业务流程图
- 3.3 数据流程图
- 3.4 技术架构拓扑图
- 3.5 关联模块具体方案描述
- 3.6 异常和边界场景
- 3.7 方案劣势、风险和解决措施

### 4. 编写技术方案

按下面原则生成内容：

- 用 Markdown 输出完整文档。
- 不要保留模板里的斜体示例内容；将示例替换为当前 feature 的实际内容。
- 对缺失信息，不要编造。使用“待补充”“待确认”并说明需要用户补什么。
- 方案描述要落到具体工程实现：涉及哪些系统、服务、模块、接口、数据结构、状态、任务、部署单元和兼容处理。
- 业务流程图优先用 Mermaid `flowchart` 表达用户或业务视角流程，并标注需求完成前后的业务影响。
- 数据流程图优先用 Mermaid `sequenceDiagram` 或 `flowchart` 表达数据流转、接口调用、事件发布/消费和状态变化。
- 技术架构拓扑图优先用 Mermaid `flowchart` 表达服务、前端、网关、数据库、缓存、消息队列、第三方系统和基础设施之间的关系。
- 关联模块具体方案描述应按实际系统拆分，例如 `前端/客户端`、`后端服务`、`数据层`、`基础设施/第三方依赖`；没有涉及时也保留标题并写“本期不涉及/待确认”。
- 异常和边界场景必须使用表格，列为：`异常情况 | 结果 | 描述 | 验证方式/次数`。如果还没验证，结果写“待验证”，验证方式或次数写计划内容或“待补充”。
- 风险部分要成对给出：风险/劣势 + 影响 + 解决措施/兜底方案 + 验证方式。

### 5. 输出后的说明

文档输出后，简要说明：

- 哪些章节已经基于资料填写。
- 哪些章节仍为待补充/待确认。
- 如果用户希望保存为文件，建议文件名：`{feature名称}_技术方案.md`。

## 输出格式

默认直接输出完整 Markdown 文档。若用户指定保存路径，则写入该路径，并在完成后说明文件绝对路径。

文档标题建议使用：

```markdown
# {Feature 名称} 技术方案
```

若用户明确要求遵循其个人 Markdown 偏好，可改为：

```markdown
文章标题：{Feature 名称} 技术方案
```

然后正文从一级标题开始。

## 质量检查清单

输出前检查：

- 模板必填章节是否全部保留。
- PRD、Jira、Confluence、接口文档是否有链接或占位。
- 三类图是否都提供，且能表达业务流、数据流和技术拓扑。
- 关联模块是否覆盖前端/客户端、后端服务、数据层、基础设施/第三方依赖等实际相关部分。
- 异常边界是否可验证，不只是泛泛而谈。
- 风险是否有对应解决措施和验证方法。
- 缺失信息是否标注为待补充，而不是猜测。
